Lock

Что это

  • В отличие от блока синхронизации, ReentrantLock дает расширенный интерфейс для получения информации о состоянии блокировки.
  • Кроме обычного ожидающего lock(), вариант tryLock() с параметром ожидает своей очереди только заданное время, а без параметра — вообще не ждет, а только захватывает свободный лок.
  • Cвойство fair - обеспечивает «справедливость» очереди: пришедший раньше поток захватывает объект раньше. Блок synchronized не дает никаких гарантий порядка.

Основные разделы

  • ReentrantLock
  • ReadWriteLock
  • StampedLock
  • Отличие от Synchronized Block

ReentrantLock

Reentrant(повторный вход) - один и тот же поток может захватить lock несколько раз.

lock.lock();
lock.lock();
//Но тогда нужно дважды unlock

Пример:

Lock lock = new ReentrantLock();

lock.lock();
try {
    counter++;
} finally {
    lock.unlock();
}
  • Можно попытаться получить lock (tryLock).
  • Можно задать timeout.
  • Можно прерывать ожидание lock.
  • Более гибкое управление.

ReadWriteLock

Интерфейс, который позволяет разделять доступ к ресурсу между потоками для чтения и записи.

  • read lock — несколько потоков могут читать одновременно, пока ни один поток не пишет.
  • write lock — только один поток может писать, блокирует всех остальных читателей и писателей.

Много потоков могут читать одновременно

Писать может только один поток, и при этом чтение блокируется

ReentrantReadWriteLock - самая популярная реализация.

  • Множественные читатели(несколько потоков с блокировкой на чтение, не мешаю друг другу). Во время чтения изменять нельзя.
  • Один писатель
  • Реентерабельность(поток может взять lock повторно)

Когда использовать

  • Твой код часто читает разделяемые данные.
  • Записи происходят редко.
  • Ты хочешь выжать максимум параллелизма: разрешить нескольким читателям работать одновременно, но гарантировать, что запись идет строго эксклюзивно.

StampedLock

Главная идея - optimistic read.

В отличии от ReadWriteLock имеет дополнительный "long stamp = lock.tryOptimisticRead();"

При optimistic read lock вообще не ставится.

Нет reentrancy

StampedLock lock = new StampedLock();

long stamp = lock.readLock();
try {
    readData();
} finally {
    lock.unlockRead(stamp);
}

Отличие от Synchronized Block

  • Synchronized Block полностью содержится внутри метода. Lock использует операции lock() и unlock() API блокировки в отдельных методах.
  • Synchronized Block не поддерживает справедливость. Любой поток может получить блокировку после снятия, и никакие предпочтения не могут быть указаны. Мы можем добиться справедливости в API-интерфейсах блокировки, указав свойство fairness. Он гарантирует, что самому долго ожидающему потоку будет предоставлен доступ к блокировке.
  • Поток блокируется, если он не может получить доступ к Synchronized Block. Lock API предоставляет метод tryLock(). Поток получает блокировку только в том случае, если он доступен и не удерживается каким-либо другим потоком. Это уменьшает время блокировки потока, ожидающего блокировки.
  • Поток, находящийся в состоянии «ожидания» получения доступа к Synchronized Block, не может быть прерван. Lock API предоставляет метод lockInterruptible(), который можно использовать для прерывания потока, когда он ожидает блокировки.
