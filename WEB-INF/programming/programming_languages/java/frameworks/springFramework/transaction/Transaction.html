Что это

Transaction - группа последовательных операций с базой данных, которая представляет собой логическую единицу работы с данными. Транзакция может быть выполнена либо полностью, соблюдая целостность данных и независимо от параллельно идущих других транзакций, либо не выполнена и должна не произвести никакого эффекта.

Основные разделы

  • Как она работает
  • Быстрый старт в Spring boot
  • Propagation
  • Transaction Isolation

TransactionManager(низкий уровень)

  • Открывает транзакцию
  • Коммитит
  • Делает rollback

Управляется в ручную, пример:

@Autowired
private PlatformTransactionManager transactionManager;

public void doWork() {
    TransactionStatus status = transactionManager.getTransaction(new DefaultTransactionDefinition());

    try {
        // бизнес-логика
        transactionManager.commit(status);
    } catch (Exception e) {
        transactionManager.rollback(status);
        throw e;
    }
}

Когда использовать:

  • Пишешь свою инфраструктуру
  • Делаешь кастомные транзакционные механизмы

TransactionTemplate(средний уровень)

TransactionTemplate — это обёртка над TransactionManager.

  • Управляет begin/commit/rollback
  • Безопаснее (rollback автоматически при exception)
  • Позволяет писать код в виде callback
@Autowired
private TransactionTemplate transactionTemplate;

public String doWork() {
    String result = transactionTemplate.execute(status -> {
        return "done";
    });
}

Когда использовать:

  • Нужен программный контроль.
  • Нельзя использовать @Transactional(например, внутри одного класса/self-invocation).
  • Нужно управлять транзакцией локально.

@Transactional(высокий уровень)

Внутри использует TransactionManager.

@Transactional - создает proxy метод.

Перед исполнением метода помеченного данной аннотацией начинается транзакция, после выполнения метода транзакция commit или при выбрасывании RuntimeException - rollback.

Под капотом создается метод с try(наш помеченный метод и commit) и catch(условия rollback).

Ошибки использования

  1. Аннотация должна быть только на public и не final методе.
  2. @Transactional можно вызывать только из другого класса, если вызвать через метод из того же класса(через метод без @Transactional), аннотация не отработает - Self-Invocation. Spring реализует @Transactional через AOP proxy. Когда внешний код вызывает твой бин, вызов идёт через прокси-объект, который оборачивает метод в транзакцию. Но когда метод вызывает другой метод того же класса через this, он обращается к реальному объекту напрямую, полностью минуя прокси.
    • Инжектировать сам себя (@Autowired self)
    • Получить прокси через AopContext
    • Разнести методы по разным бинам(лучшее решение)
  3. @Transactional - откатиться только с RuntimeException и его наследниками. Для примера с Exception class она не откатиться. Но можно указать ожидаемое исключение в параметре rollbackFor. Пример: @Transactional(rollbackFor = Exception.class)
  4. Транзакция занимает соединения с БД. По этому нужно выносить долгую логику за пределы транзакции.

Быстрый старт в SpringBoot

Указать аннотацию @Transaction над необходимым методом(в слое service).

Propagation

Propagation - определяет границу транзакции нашей бизнес-логики. Spring управляет запуском и приостановкой транзакции в соответствии с нашими настройками propagation. Spring вызывает TransactionManager::getTransaction для получения или создания транзакции в соответствии с распространением.

  1. REQUIRED(default) - если есть транзакция добавиться к ней, если нет создать.
  2. SUPPORTS - если есть транзакция добавиться к ней, если нет выполниться нетранзакционно.
  3. NOT_SUPPORTED - если есть транзакция, Spring приостанавливает ее, а затем бизнес-логика выполняется без транзакции.
  4. MANDATORY - если есть транзакция добавиться к ней, если нет выдать исключение(IllegalTransactionStateException).
  5. NEVER - если есть транзакция выдать исключение(IllegalTransactionStateException).
  6. REQUIRES_NEW - если есть транзакция Spring приостанавливает ее и создает новую.
  7. NESTED - Spring проверяет, существует ли транзакция, и если да, то отмечает точку сохранения. Это означает, что если выполнение нашей бизнес-логики выдает исключение, то транзакция откатывается к этой точке сохранения. Если активной транзакции нет, это работает как REQUIRED.

Transaction Isolation

  1. DEFAULT - использовать уровень изоляции, заданный в СУБД. Например в PostgreSQL → READ COMMITTED.
  2. READ_UNCOMMITTED
  3. READ_COMMITTED
  4. REPEATABLE_READ
  5. SERIALIZABLE

readOnly = true

readOnly = true - это подсказка для оптимизации, а не строгое ограничение.

Hibernate может пропустить flush (не синхронизировать изменения с БД)

БД может оптимизировать запрос, не создавая блокировки

Некоторые драйверы могут использовать read-only соединения
