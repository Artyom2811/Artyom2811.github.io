Spring Core

  • Дополнительные возможности ApplicationContext

Что делает

  1. Удобство создания объектов через Beans.
  2. IoC - через управление beans.
  3. IoC Container - это реализация IoC и DI. Контейнер IoC создает и управляет bean-компонентами на основе мета-информации. Он также может решать и предоставлять зависимости для создаваемых им объектов.
  4. BeanFactory - это интерфейс, который создает и предоставляет bean-компоненты на основе BeanDefinition-ов. Он является ядром ApplicationContext.

ApplicationContext

ApplicationContext - это центральный интерфейс который предоставляет следующий список возможностей.

  • Возможности BeanFactory.
  • Загрузка ресурсов.
  • Публикация событий.
  • Интернационализация.
  • Автоматическая регистрация BeanPostProcessor и BeanFactoryPostProcessor.

Основные разделы

  • Виды Конфигурации beans

BeanFactory

BeanFactory - корневой интерфейс обеспечивает базовую поддержку внедрения зависимостей.

  • Ленивая инициализация - бины создаются только при вызове getBean().
  • Минимальный расход ресурсов.
  • Нет поддержки AOP, событий, i18n и аннотаций «из коробки».

ApplicationContext

Расширяет BeanFactory.

  • Жадная инициализация — все singleton-бины создаются при старте.
  • Встроенная поддержка:
    • AOP
    • Аннотации @Autowired, @Component, @PostConstruct и др.
    • Автоматическая регистрация BeanPostProcessor/BeanFactoryPostProcessor
    • Environment — профили и источники свойств
    • MessageSource - интернационализация
    • ApplicationEvents — публикация и подписка на события

Имплементации:

  • ClassPathXmlApplicationContext - загружает XML из classpath.
  • FileSystemXmlApplicationContext - загружает XML из файловой системы(напрямую с диска).
  • AnnotationConfigApplicationContext - использует аннотации(@Configuration, @Bean, @ComponentScan и др.)
  • GenericGroovyApplicationContext - это способ конфигурировать Spring через Groovy DSL.

WebApplicationContext

WebApplicationContext - расширение для ApplicationContext для веб-приложений. Используется в Spring MVC.

Имена в spring контексте

  1. При создании через @Component, etc - имя bean будет использовано имя аннотированного класса.
  2. При создании через @Bean - имя bean будет использовано имя аннотированного метода.

Виды конфигурации beans

  1. XML конфигурация - ClassPathXmlApplicationContext или FileSystemXmlApplicationContext.
  2. Аннотационная конфигурация - AnnotationConfigApplicationContext. Через @ComponentScan находятся @Component, @Service, @Repository или @Controller. Для inject используется @Autowired.
  3. Java Config - AnnotationConfigApplicationContext. @Configuration + @Bean. Это как XML только в JAVA, без @Service и @Inject и тд., только @Bean. Используется когда мы используем сторонние библиотеки и не можем использовать на них аннотации.
  4. Groovy конфигурация — GenericGroovyApplicationContext(“context.groovy”).

Какой выбрать?

В современных проектах (Spring Boot) обычно комбинируют Java-конфигурацию + аннотации:

  • @Service/@Repository - для своих классов.
  • @Bean в @Configuration - для сторонних библиотек, где нельзя поставить аннотацию на класс.

Виды Dependency injection(DI) в порядке выполнения JAVA:

  1. Через конструкторы - лучший способ.
    • Поля final.
    • Предотвращает Circular Dependency.
    • Упрощает модульное тестирование, позволяя подставлять mocks или stabs через конструктор.
    • Все зависимости видны в конструкторе, что чётко показывает, что нужно классу для работы.
    • Внедрение через конструктор не использует рефлексию или доступ к приватным полям (в отличие от внедрения через поля), что повышает производительность и снижает риски.
  2. Через поля - плохая практика.
  3. Через сеттеры - плохая практика.

Без @Autowired

Начиная с Spring Framework 4.3 если внедряемый класс через конструктор, имея один конструктор, @Autowired можно не использовать.
